Most release notes tell you what shipped. Fewer tell you why the product had to take this shape.

That is the better question for BisQue Ultra. We did not build a chat interface and then go looking for scientific uses. We started with the opposite problem: research software already had storage, viewers, metadata, modules, and serious analysis tools, but too much of that power was still difficult to inhabit. The system could do the work. The product surface still made the work feel heavier than it needed to.

BisQue Ultra is our answer to that gap.

It is not a replacement for BisQue. It is a new working surface built on top of BisQue. The distinction matters. BisQue already knows how to store large scientific data, visualize up to 5D imagery, carry flexible metadata, and host domain-specific modules. Throwing that away would have been wasteful. The real task was to make those strengths easier to read, easier to steer, and easier to explain after the fact.

<Callout title="The short version" tone="proof">
  BisQue Ultra tries to make scientific work feel lighter without making it less inspectable.
</Callout>

## Start with the platform, not the interface

The most important architectural decision was the earliest one: BisQue Ultra would be built on top of the BisQue platform rather than beside it.

That choice sounds obvious once it is stated, but it carries a real philosophy. In research software, novelty is often overrated. The harder discipline is continuity. If a platform already supports image storage, visualization, metadata, annotations, and analysis modules, then the next product should preserve that foundation and improve the path through it. A new surface is useful only if it keeps the old scientific guarantees alive.

That is why BisQue Ultra still treats core objects as first-class realities:

- uploaded files
- BisQue resources and dataset imports
- multidimensional image viewers
- run records
- generated artifacts
- reproducibility outputs

In other words, the product is not trying to abstract science away. It is trying to make the scientific objects easier to follow.

## The stack in one sentence

BisQue Ultra uses a React frontend, a FastAPI backend, a lean Agno runtime, the BisQue platform as its scientific substrate, and an OpenAI-compatible model path that currently defaults to `vLLM` serving `gpt-oss-120b`.

That stack is not eclectic. It is practical. Each layer was chosen because it answers a specific problem in the product.

## Why the frontend is React and Vite

The frontend had to do two things at once. It had to feel calm, and it had to stay honest about state.

Those goals rule out a surprising number of fashionable approaches. A scientific interface is not a static landing page. It is a moving system. A user may be uploading data, streaming a response, opening a viewer, switching a workflow preset, reading a result card, or waiting on a long-running task. The interface has to show all of that without becoming noisy.

React is useful here because it is explicit. State has a place to live. Conditional surfaces have a language. The UI can be decomposed into pieces that are readable in code and understandable in behavior. That matters when the product has to juggle chat, tools, viewers, uploads, and artifacts in one shell.

Vite answers a different problem. It keeps the development loop fast enough that the team can keep refining small details. That may sound secondary, but it is not. A calm interface does not appear in one pass. It emerges through repeated comparison: this weight or that one, this hover or that one, this composer height or that one. A slow development loop encourages rough decisions to stay rough. A fast one makes polish cheaper.

The surrounding frontend choices follow the same logic. TypeScript makes contracts harder to hand-wave away. Playwright turns interaction quality into something we can test instead of merely hope for. Inter and JetBrains Mono give the interface two clear voices: one for reading, one for technical evidence.

The result is not a frontend that looks busy. It is a frontend that can afford to stay quiet because its state is doing real work underneath the surface.

## Why the backend is FastAPI

A scientific product does not only need a backend. It needs a boundary.

FastAPI is useful in BisQue Ultra because the product is really a set of contracts. There is a contract for chat. A contract for uploads. A contract for resumable uploads. A contract for run events. A contract for artifacts. A contract for previews and reports. If those shapes drift, the product becomes harder to trust, harder to debug, and harder to evolve.

FastAPI gives us a clean place to define that boundary. The frontend does not need to guess what a run looks like. The runtime does not need to leak its internal structure into the browser. The upload layer does not need to become a private dialect known only to one UI component.

That boundary matters even more in scientific software than it does in ordinary software. Research teams need to revisit old work. They need to compare results across time. They need to understand what happened when a workflow changed. Stable contracts are not just engineering hygiene. They are part of reproducibility.

## Why the runtime is Agno, but lean

It is easy to build an agent system that sounds impressive. It is harder to build one that behaves predictably.

BisQue Ultra uses Agno, but it uses Agno with restraint. The default chat path stays single-agent by default. Tool use is optional and allowlisted. Mutating actions are approval-aware. Runtime budgets are enforced in code. Run events are persisted as machine-readable records rather than hidden inside assistant prose.

That design is deliberate.

The temptation in agentic systems is to solve uncertainty by adding more agents, more hidden debates, more planner layers, more internal performance. Sometimes that helps. Often it makes the product harder to understand. For a scientific interface, that trade is dangerous. A user should not have to wonder whether the system took one clean reasoning pass or five opaque internal detours.

So we chose a stricter default: one direct model pass when one direct model pass is enough. If tools are needed, the system exposes that. If approval is needed, the system pauses. If a run produces artifacts, those artifacts are saved. The runtime is allowed to be powerful, but it is not allowed to become mysterious.

<Callout title="Runtime principle">
  Capability is useful only when the product can still explain what happened.
</Callout>

## Why we route through vLLM

The model path matters because it shapes both operations and product design.

BisQue Ultra is configured to support multiple providers. The code can route through `vllm`, `openai`, or `ollama` using the same OpenAI-compatible client posture. That flexibility is not accidental. It keeps the application from being welded to one vendor or one deployment model.

But the default path in this project is `vLLM`, and that is an important choice.

We chose `vLLM` because it gives the system a model server we can host and reason about while still speaking a familiar API. That matters in a research setting for several reasons:

- it keeps deployment closer to lab infrastructure
- it reduces the amount of application code that has to change when the model path changes
- it makes it easier to evaluate open-weight models without redesigning the product contract
- it gives the team more control over latency, scaling, and cost than a purely external dependency would

There is also a subtler reason. An OpenAI-compatible interface is not just convenient. It is a portability layer. It lets the product remain architecturally stable while the model underneath it evolves.

That stability matters more in science than it does in ordinary software. Research teams often keep a workflow alive for months. They cannot afford an application whose contracts change every time the serving layer does. `vLLM` gives us a way to keep the product steady while still evaluating new models, new serving hardware, and new deployment tradeoffs behind the curtain.

## Why the flagship path is gpt-oss-120b

Once the serving path was chosen, the harder question was the model itself.

The repository keeps lighter options available, including local Ollama defaults for smaller setups. That is the right fallback posture. Not every environment needs the same model footprint.

But the flagship path is `gpt-oss-120b`, and that choice says something specific about the product. BisQue Ultra asks the model to do more than chat pleasantly. It asks the model to read scientific context, keep track of tool opportunities, respect structured outputs, produce long answers that remain coherent, and move between explanation and evidence without falling apart. A smaller model can be useful. It can even be surprisingly good. But the default experience should not feel one clever prompt away from collapse.

What made `gpt-oss-120b` persuasive was not only its size. It was the shape of the trade. OpenAI describes it as a 117B-parameter mixture-of-experts model with 5.1B active parameters per token, 36 layers, 128 experts with 4 active per token, and a native 128k context window. That combination matters. It means the model has large-model headroom without forcing every token to pay the full cost of a dense 117B pass. In practical terms, that gives BisQue Ultra a model that can stay ambitious without becoming absurd to serve.

Just as important, the model is not merely large. It is strong in the places scientific software actually needs strength.

- On **GPQA Diamond**, OpenAI reports `67.1` at low reasoning, `73.1` at medium, and `80.1` at high without tools; with tools, the score rises to `80.9`.
- On **MMLU**, the model climbs from `85.9` to `88.0` to `90.0` as reasoning effort increases.
- On **SWE-Bench Verified**, it moves from `47.9` to `52.6` to `62.4`.
- On **Tau-Bench Retail**, a useful proxy for multi-step tool use, it rises from `49.4` to `62.0` to `67.8`.

Those numbers are worth pausing over. They tell us two things.

First, the model has real depth. GPQA is not a trivial benchmark. It is a graduate-level science benchmark designed to punish shallow pattern matching. An `80.1` score without tools, and `80.9` with tools, means the model is not merely fluent. It can hold onto hard scientific questions long enough to answer them well.

Second, the model rewards extra thought. OpenAI’s own model card shows smooth gains as reasoning effort moves from low to medium to high. That is not a minor convenience. It is a design opportunity.

BisQue Ultra takes advantage of that opportunity directly. The model supports low, medium, and high reasoning effort. Our own API exposes that same idea in product language through `fast`, `auto`, and `deep` reasoning modes. The point is not to make every turn slower and “smarter” by default. The point is to give the system gears.

If a user needs a quick routing decision, lightweight intake, or a simple factual answer, the system can stay fast. If the task needs synthesis, verification, or careful tool orchestration, the runtime can spend more reasoning budget where it actually pays off. In Pro Mode, for example, the intake phase is intentionally kept lighter, while longer discussion, synthesis, and verifier phases are allowed more time. That is a better use of intelligence than forcing one fixed tempo on every request.

<Callout title="The real superpower">
  The advantage of `gpt-oss-120b` is not only that it knows more. It is that the model can think in different gears, and BisQue Ultra is built to choose the right one for the task.
</Callout>

There is one more reason this model fits the product. OpenAI says the `gpt-oss` family was post-trained with the same general playbook used for `o4-mini`, including supervised fine-tuning, a high-compute RL stage, explicit chain-of-thought reasoning, and tool use before the answer. That matters because BisQue Ultra is not asking for bare text generation. It is asking for an assistant that can reason, decide when tools are relevant, and still return a readable answer after the tools have spoken.

So the choice was not “big model versus small model.” The real choice was between a model that merely responds and a model that can become a scientific instrument. `gpt-oss-120b` gave us open-weight control, a portable serving path, a 128k working memory, strong GPQA performance, and a reasoning ladder we could expose productively. That is why it became the flagship path.

## Why runs, artifacts, and upload states are part of the product

Many AI interfaces behave as if the answer is the product and everything else is scaffolding. That model breaks down quickly in scientific work.

In research, the answer is only the beginning of the conversation. The more important questions come right after it:

- Which file did this come from?
- Which run produced it?
- Which workflow was active?
- Which artifacts were generated?
- Can I recover the evidence later?

That is why BisQue Ultra treats uploads, run events, and artifacts as first-class objects. Uploads move through explicit lifecycle states. Run metadata is persisted. Artifacts live under run-scoped directories with manifests and source references. Event payloads stay machine-readable so the interface does not have to guess what happened by parsing decorative text.

This is not implementation trivia. It is the difference between a result that feels impressive for ten seconds and a result that can survive scrutiny next week.
