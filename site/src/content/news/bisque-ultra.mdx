Today we are introducing BisQue Ultra, a new scientific work surface built for researchers who need image analysis, model reasoning, and reproducible outputs to stay in the same field of view.

The BisQue platform already knew how to store scientific data, visualize up to 5D imagery, preserve metadata, and run domain-specific analysis. What was still missing was a product surface equal to that foundation: one that made serious work easier to start, easier to steer, and easier to explain after the fact.

BisQue Ultra is our answer to that gap. It does not replace BisQue. It makes BisQue easier to inhabit.

## What BisQue Ultra actually is

BisQue Ultra is a working surface built on top of the BisQue platform. That distinction matters because the product was never meant to discard the scientific substrate and start over with a prettier chat box. The point was to keep the hard-won capabilities that already matter in research, then make them easier to read and easier to trust.

That is why BisQue Ultra still treats the scientific objects as first-class realities:

- uploaded files
- BisQue resources and dataset imports
- multidimensional image viewers
- run records
- generated artifacts
- reproducibility outputs

In other words, the product is not trying to abstract science away. It is trying to make the scientific objects easier to follow.

## Why the stack took this shape

BisQue Ultra uses a React frontend, a FastAPI backend, a lean Agno runtime, the BisQue platform as its scientific substrate, and an OpenAI-compatible model path that currently defaults to `vLLM` serving `gpt-oss-120b`.

That stack is not eclectic. It is practical. Each layer solves a different problem, and the product only works because those problems were separated cleanly instead of folded into one clever but fragile system.

React is there because the interface has to tell the truth about state. A scientific surface is always moving: uploads arrive, responses stream, viewers open, workflows change, and results accumulate. FastAPI is there because the system needs contracts strong enough to keep chat, uploads, run events, and artifacts legible over time. Agno is there because model reasoning needs a runtime, but the runtime is kept lean on purpose. One clean model pass when one clean model pass is enough is a better default than hidden internal theater.

## Why the flagship model path is gpt-oss-120b

Once the serving layer was chosen, the harder question was the model itself.

The flagship path is `gpt-oss-120b` because BisQue Ultra needs more than agreeable language. It needs a model that can hold scientific context, survive longer reasoning chains, decide when tools matter, and still return an answer a scientist can actually use. OpenAI describes `gpt-oss-120b` as a 117B-parameter mixture-of-experts model with 5.1B active parameters per token, 36 layers, 128 experts with 4 active per token, and a native 128k context window. That trade is what made it compelling: large-model headroom without paying the full dense-model cost on every token.

The benchmark profile matters too. OpenAI reports GPQA Diamond at `67.1` on low reasoning, `73.1` on medium, and `80.1` on high reasoning without tools; with tools it reaches `80.9`. MMLU rises to `90.0` at high reasoning. SWE-Bench Verified reaches `62.4`. Tau-Bench Retail reaches `67.8`. Those are not just nice numbers to cite. They show a model that gets better when it is allowed to think harder, which is exactly the kind of behavior a scientific product can put to work.

That is why BisQue Ultra exposes `fast`, `auto`, and `deep` reasoning modes instead of treating intelligence like one fixed speed. A quick routing decision, a short factual turn, and a long analytical synthesis should not all pay the same cognitive or latency cost. The model's reasoning ladder gives the product gears. That is the real advantage. Scientists do not only get a stronger model. They get a system that can choose when to think lightly and when to lean in.

## Why BioIO sits at the image boundary

One quiet choice matters more than it first appears: BisQue Ultra uses BioIO, the successor to AICSImageIO from the Allen Institute for Cell Science image-I/O line. That decision lives below the chat surface, but it shapes the honesty of everything above it. Scientific files are not ordinary pictures. They arrive as scenes, channels, z-stacks, time series, mosaics, and metadata-heavy containers that punish any interface willing to guess. BioIO gives us a stricter first contact. Its `BioImage` interface reads microscopy data into a consistent 5D `TCZYX` model, exposes scenes, dimensions, channel names, metadata, and physical pixel sizes, and can stay lazy through Dask until a real chunk needs to be pulled into memory. That is exactly the posture we wanted: inspect structure first, spend compute second.

The package is also broad enough to keep the product unified instead of turning file support into a patchwork of exceptions. The official BioIO docs list 12 maintained reader plug-ins covering `CZI`, `DV`, `LIF`, `ND2`, `OME-TIFF`, tiled `OME-TIFF`, `OME-Zarr`, `SLDY`, `TIFF`, globbed `TIFF`, and imageio-backed formats such as `PNG` and `GIF`, plus a `Bio-Formats` bridge for many more microscopy files. In this repository we install the base `bioio` package alongside 11 reader plug-ins, including `bioio-czi`, `bioio-nd2`, `bioio-ome-tiff`, `bioio-ome-zarr`, `bioio-tifffile`, and `bioio-bioformats`. That is not overengineering. It is how one product surface can open real scientific data without asking the user to care which lab instrument wrote the file.

That choice shows up directly in the code. Our imaging layer is described as a universal entrypoint for scientific image loading and preview generation, and the `bioio_load_image` tool uses that layer to normalize dimensions, surface previews, and recover metadata before heavier reasoning begins. In practice, BioIO helps BisQue Ultra answer the questions that matter at the start of scientific work: What is this file? How many channels does it have? Is it a volume, a time series, a mosaic, a scene collection? What are the physical scales? A good scientific assistant should not start with improvisation. It should start by reading the specimen correctly.

## What becomes easier now

The launch matters because a few workflows get meaningfully easier at once:

- You can import a scientific file and start from its real structure rather than from a guessed image preview.
- You can ask for a report, a detection run, or a follow-up explanation without losing the link back to the source data.
- You can move between language, viewer context, run status, and generated artifacts without treating each one as a separate tool.
- You can return later and still recover what happened, what file it happened on, and what outputs the run produced.

Those are small sentences, but they point to a larger change. BisQue Ultra is trying to lower the cost of serious work without lowering the standard of evidence.

## Why runs, artifacts, and upload states are part of the product

Many AI interfaces behave as if the answer is the product and everything else is scaffolding. That model breaks down quickly in scientific work.

In research, the answer is only the beginning of the conversation. The more important questions come right after it: Which file did this come from? Which run produced it? Which workflow was active? Which artifacts were generated? Can I recover the evidence later?

That is why BisQue Ultra treats uploads, run events, and artifacts as first-class objects. Uploads move through explicit lifecycle states. Run metadata is persisted. Artifacts live under run-scoped directories with manifests and source references. Event payloads stay machine-readable so the interface does not have to guess what happened by parsing decorative text.

This is not implementation trivia. It is the difference between a result that feels impressive for ten seconds and a result that can survive scrutiny next week.
